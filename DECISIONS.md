## Step 1 — Scaffold

This matches the approved directory tree exactly (`core/contracts/`, `core/detect/`, ... directly under the workspace-member directory) without introducing a redundant nested `core/core/` src-layout directory. Likewise `gateway/`'s pieces (`app`, `config`, `api`, `auth`, `streaming`, `router`, `providers`) are flat top-level modules. This mirrors how the pre-rewrite codebase already resolved `monoai_gateway`/`monoai_router`/`pii_pipeline` as flat top-level packages.
- **Secrets stage split from regex stage.** `SENTINEL-2.0/pii_pipeline/rampart/regex.py`'s `_detect_secrets`/`_SECRET_PATTERNS`/`_PASSWORD_CUE_RE` move to their own `core/detect/stages/secrets_stage.py` rather than staying fused inside `regex_stage.py`, so secrets detection gets its own detector-version string for audit attribution (a sovereignty-relevant surface worth tracking independently).
- **Hash-chaining implemented in Phase 1, not deferred.** `core/audit/chain.py`'s `compute_hash`/`append`/`verify` are real in Phase 1 (cheap to implement, and invariant #2 — "chain to the previous record's hash" — applies to all of Phase 1). Only the Postgres/webhook sinks and the Ed25519-signed evidence export layer defer to Phase 2.
- **SQLite before Postgres, for both vault and auth key storage.** `core/vault/storage/postgres_store.py` and a `PostgresKeyStore` in `gateway/auth/` are stubs (`NotImplementedError`) in Phase 1; SQLite backs both. Rate-limit counters use Valkey (already a hard dependency for the vault master key), reused under a separate key namespace — no new infra introduced for Phase 1.
- **ONNX NER model binary copied into `core/detect/packs/base_en/models/` (~66MB), tracked in git for now.** Preserves existing `PII_USE_ONNX_NER=true` functionality. Should move to a checksummed fetch-at-build-time artifact (`scripts/get_models.sh`) in Phase 2, consistent with the `ner-sidecar` model-download plan — flagged here so it isn't forgotten.
- **Rate limiter uses plain GET/SET, not a Lua EVAL script.** `gateway/auth/rate_limit.py`'s token bucket reads then writes bucket state as one JSON blob per key, rather than an atomic Lua script. Under truly concurrent bursts to the same key at the same instant there's a small race window; the sequential/fake-clock proof tests this phase ships are unaffected. A Lua-atomic version is a flagged hardening item, not implemented this phase.
- **Negation override moves from detection to policy.** `locked_spans.py`'s old behavior of setting `override_classification="PRESERVE"` directly is replaced: `core/detect/stages/locked_span_stage.py` now only attaches `span.meta["negated"] = True`; the actual PRESERVE override is applied by `core/policy/engine.py`'s `overrides.locked_span_negation` rule, per the mission's explicit instruction that action-assignment moves out of detectors into policy.

## Step 9 — Streaming, chat API, orchestrator rebuild

- **Difficulty classification strips PII tokens first.** The hex-based token format (needed for G8's value-determinism) contains embedded digit runs that spuriously tripped `classify_difficulty`'s "math word problem" heuristic (≥3 digit groups), silently reclassifying simple PII-bearing prompts as "moderate". Fixed by replacing `[PII_TOKEN_xxxxxxxxxx]` with a neutral placeholder before classification only (the real tokens reach the provider unchanged) — see `PiiEngine.strip_tokens_for_classification`.
- **Output-scan protects existing token ranges from re-detection, and skips NER on small windows.** Two real bugs found during manual end-to-end testing: (1) the real ONNX NER model can misclassify `[PII_TOKEN_xxxxxxxxxx]` bracket+hex syntax as `NEARBYGPSCOORDINATE`, corrupting a legitimate placeholder; (2) on tiny out-of-context streaming flush-window fragments (e.g. `"mple"`, a mid-word remainder of `"[simple]"`), the same model hallucinates entities from a handful of characters. Fixed by (1) excluding spans overlapping an existing complete token from `scan_output`'s decisions, and (2) adding `include_ner: bool` down through `DetectionPipeline.run`/`PiiEngine.scan_output`, which `gateway/streaming.py` sets to `False` for per-chunk scans (regex/secrets detectors remain on and are reliable at any window size).
- **Streaming's sliding-window split is content-aware, not a fixed-width slice.** An initial version held back a flat `TOKEN_LEN - 1` characters per flush, which (a) had an off-by-one that clipped the leading `[` off a just-completed token, and (b) still fragmented a token across 3+ separate `_process()` calls under small (4-char) chunk granularity, since each call only saw a slice too small to regex-match a whole token. Fixed: `StreamRehydrator._split` finds the last `[` before the naive split point and either pulls the boundary back (incomplete/ambiguous token) or pushes it forward past a complete token that would otherwise straddle the cut — a token is always handed to `_process()` as one whole string exactly once.
- **Streaming endpoint chunks a non-streaming provider response artificially** (`gateway/api/chat.py`'s `_artificial_upstream`, 4 chars at a time) rather than proxying real upstream SSE from the provider — `StubProvider`/`OpenAICompatibleProvider`/`OllamaProvider` only implement one-shot `complete()`. The sliding-window rehydrator itself is fully real and independently tested against arbitrary chunk boundaries; true token-by-token upstream streaming is a later refinement.
- **AuditChain resumes from the existing log's last hash on startup.** A fresh `AuditChain(initial_last_hash=None)` appending to an already-populated jsonl file (e.g. after any process restart) would write a record whose `prev_hash=None` doesn't match the file's real last hash, breaking the chain — found via manual testing across repeated dev-server restarts. `gateway/app.py`'s lifespan now calls `audit.sinks.read_last_hash(path)` to bootstrap it correctly.
- **gateway/router/ absorbs monoai_router's contracts+normalizer as a Step 9 prerequisite**, not deferred to a later step as the original per-step plan text implied — `gateway/providers/` needs `RequestContext`/`Message`/`ProviderResponse` to have something to operate on, so these were pulled forward from their originally-implicit "whenever needed" placement into Step 8/9 rather than invented ad hoc.

## Phase 1 — Stabilization & fix

- **Vault TTL cache bug: cache hits never re-checked `expires_at`.** `SqliteVaultStore.get()`/`PostgresVaultStore.get()` both returned a cached plaintext value unconditionally on a cache hit, only enforcing `expires_at` on the cold-path DB row. Since `write_async` populates the in-memory cache immediately and nothing ever evicted it, a token written with a short TTL stayed live in the process cache forever (or until a restart) even after the physical row was eligible for `sweep_expired()` — a real sovereignty/erasure gap, not just a performance quirk. Fixed by storing `(plaintext, expires_at)` tuples in `_cache` instead of bare strings, checking `expires_at` against `time.time()` on every hit (cache or DB), and evicting the cache entry the moment it's found stale rather than waiting for the next `sweep_expired()` pass. `write_async` now computes `expires_at` once at call time (not later inside the executor thread) so the cached copy and the durable row always agree on the same expiry instant.

## Phase 2 — Governance depth & routing intelligence

- **G4 semantic injection judge lives in `core/detect/stages/injection_judge.py`, not gateway/.** It makes real network calls (Ollama, optionally Claude API), but `core/audit/sinks.py`'s `WebhookSink` already established that `core/` can own I/O when it's the *detection/audit* concern itself, not application/service wiring -- same precedent applies here. The module takes plain constructor args (base URL, model name, API key), never a gateway `Settings` object, so `core/` stays free of a `gateway` import. Trigger logic (`should_invoke_judge`) checks the Tier 2 detector's confidence against a `[0.4, 0.8]` ambiguous band OR `policy.injection.semantic_depth`; any judge failure (unreachable Ollama, no Claude key, bad/unparseable output) returns `available=False` and `gateway/pii.py::_check_injection` falls back to the Tier 2 verdict -- an LLM outage must never block the request path.
- **Claude fallback only fires if `claude_api_key` is configured; there's no attempt to detect "Ollama is down forever" vs. "this one call timed out."** Keeping it stateless (retry Ollama fresh on every call, no circuit breaker) was a deliberate scope cut -- G6's `CircuitBreaker` exists and could be reused later, but doing so now would mean sharing state across every `PiiEngine` instance for a judge that's already an optional, fail-open upgrade.
- **G7's `gateway/router/embedding_router.py` supersedes `embedding_classifier.py`'s ad hoc single-TF-IDF "G7" labeling (see that file's own docstring) without deleting or modifying it** -- `tests/unit/test_router_cascade.py` exercises it directly, and Phase 2's instruction was "skip tests," not "break tests." The new module is the real dataset-size-gated cascade (anchor similarity <50, "setfit" 50-299, "onnx_distilbert" >=300 examples in `bench/corpora/routing_labeled.jsonl`) with per-decision confidence + rationale, wired into `Orchestrator` as an additional confirm-or-override stage after the existing heuristic -> G6-classifier cascade (only overrides when `confidence >= 0.4`, fail-open to the earlier stages otherwise).
- **G7's "setfit" and "onnx_distilbert" tiers are not real SetFit/sentence-transformers or a fine-tuned ONNX DistilBERT model -- user-confirmed substitution.** Asked directly (real deps: `setfit`+`sentence-transformers`+`torch`, multi-GB install + HF Hub downloads, vs. a lightweight TF-IDF+sklearn substitute consistent with the codebase's existing precedent for `injection_stage.py`/`embedding_classifier.py`); user chose the lightweight path. Both tiers are real, trained-at-construction scikit-learn models (word 1-2gram TF-IDF+LogisticRegression for "setfit", char 3-5gram TF-IDF+LogisticRegression for "onnx_distilbert", approximating a subword tokenizer's finer-grained signal) behind the exact interface a real SetFit/ONNX model would expose, so swapping the real libraries in later only touches that one file.
- **G9's `gateway/obs/otel.py` is new, additive to `tracing.py`/`metrics.py`, not a replacement.** `tracing.py` keeps owning the `TracerProvider`/`stage_span` machinery already used everywhere; `otel.py` adds a real OTLP/HTTP exporter path (`opentelemetry-exporter-otlp-proto-http`, new gateway dependency) selected by `OTEL_EXPORTER_OTLP_ENDPOINT`, degrading to `ConsoleSpanExporter`/`ConsoleMetricExporter` when absent, plus four new OTEL metric instruments (request latency, token consumption, PII density, rehydration overhead) that are no-ops until `configure_otel()` has run -- existing Prometheus metrics in `obs/metrics.py` are untouched and still serve `/metrics`. `gateway/app.py`'s module-level `configure_tracing()` call is replaced by `configure_otel(...)` reading settings once at import time. New `stage_span` coverage added for "auth" (`gateway/api/chat.py`), "detect"/"policy"/"vault" (`gateway/pii.py`, nested inside the existing "sanitize" span), and "streaming_rehydrate" (`gateway/streaming.py`) -- "provider" and "audit" were already instrumented.
- **`uv sync` (no flags) wipes the workspace venv down to just the root project's (empty) dependencies + dev-group tools.** This repo's root `pyproject.toml` is a "virtual" workspace root with `dependencies = []`; plain `uv sync` only installs what the root project itself depends on, which is nothing, silently uninstalling fastapi/opentelemetry/scikit-learn/etc. from a previously-populated venv. Discovered the hard way after adding the OTLP exporter dependency. **Always use `uv sync --all-packages`** in this repo to (re)install/update the shared venv across both workspace members.
- **G13's per-record signer lives in `core/audit/signer.py`, deliberately separate from `core/audit/signing.py` (evidence-bundle key, G10) and `core/audit/evidence.py` (whole-bundle signature, G10).** G13 signs every individual chained record at append time (not just an exported bundle) so a single tampered/replayed JSONL line is detectable in isolation. Key derivation is a deterministic SHA-256 of `MONOAI_AUDIT_SIGN_KEY` (or, if unset, `SESSION_TOKEN_SECRET`) rather than the Valkey-stored load-or-create pattern `signing.py` uses for the evidence key -- deliberately so every gateway worker process derives the identical signing key from shared config alone, without a Valkey round trip on every record append (append is on the hot path; evidence export is not).
- **`compute_hash`'s canonical JSON now excludes `signature` alongside `hash`.** The new `AuditRecord.signature` field is set *after* `compute_hash` runs (signature covers the hash, so it can't also be an input to it) -- without also excluding it from the canonical dump, every signed record's hash would fail to reproduce on read-back (computed once with `signature=None`, recomputed later against the persisted non-None value). Excluding a field that's always `None` for pre-existing (unsigned) records is a no-op for old chains, so this is backward compatible.
- **Reading-side enforcement ("reject unsigned entries") is opt-in per call (`require_signature` param on `read_jsonl`/`PostgresSink.read_all`), not automatically applied everywhere `MONOAI_AUDIT_SIGN=true`.** `gateway/api/evidence.py`'s export endpoint and `scripts/verify_audit_chain.py` (new `--require-signed`/`--public-key` flags) both pass it through from settings/CLI args; `gateway/app.py`'s own startup bootstrap read (just extracting the last hash to resume the chain) does not, since flipping the flag on a deployment with pre-existing unsigned history would otherwise hard-crash every restart -- a call the user should make explicitly (e.g. via a one-time migration/re-sign pass), not one this phase silently forces.

## Phase 3 — Multi-surface scanning & streaming

- **filescan-worker/ is a new, standalone uv workspace member (`monoai-filescan-worker`), not a route bolted onto the main gateway.** `gateway/api/files.py`'s existing `/v1/files/scan` stub already deferred to "see filescan-worker/README.md" as a separate service, so the new `filescan-worker/app.py` runs as its own FastAPI process (`uv run uvicorn app:app --app-dir filescan-worker`) that directly imports `monoai-core`'s `detect`/`policy` packages -- no gateway/vault/session dependency. `TextUnitLocator.surface` gained `"file_field"` (and `"mcp_arg"` for Phase 4) per its own docstring's forward-looking note.
- **File scanning has no vault/rehydration story.** A dropped document isn't a chat session -- there's no `session_id` to key a vault entry on, and no later turn to rehydrate into. `filescan-worker/scan.py` masks both BLOCK and REVERSIBLE spans identically in its report (`[REDACTED_<LABEL>]`); the value is proof of what was found and where, not a reversible placeholder.
- **PDF/DOCX/XLSX text extraction uses pypdf/python-docx/openpyxl (all pure-Python-ish, no heavy native/ML deps)** -- a normal, small dependency addition (unlike G7's SetFit/torch tradeoff), added without a sign-off round trip. CSV needs no extra dependency (stdlib `csv`). Each extracted unit (PDF page / DOCX paragraph+table row / XLSX sheet row / CSV row) becomes its own `TextUnit`, so a finding's `location` in the report points at roughly where in the document it lives.
- **scan-async uses an in-memory `ThreadPoolExecutor`-backed job store (`filescan-worker/jobs.py`), not Valkey/Postgres/Celery.** Consistent with core/vault's own Phase 1 SQLite-before-Postgres precedent: real, correct, single-process behavior now; multi-worker-safe job tracking flagged as a later hardening item rather than pulled into scope for what the brief calls a "lightweight module."
- **G5 output-scan context suppression (`OutputScanConfig.suppress_context`) blanks whole sentences adjacent to (not containing) an actioned span, via `core/detect/sentence_split.split_sentences`.** The sentence that actually contains the PII match keeps the normal per-span token/redaction substitution (unchanged); only its immediate left/right neighbor sentences get replaced wholesale with `[CONTEXT_SUPPRESSED]`, on the theory that neighboring sentences often still identify someone even once the literal token is gone (e.g. "She lives in Boston." right after a masked name). Only triggers off BLOCK/REVERSIBLE decisions, not PRESERVE.
- **Known interaction: an already-tokenized `[PII_TOKEN_xxxxxxxxxx]` placeholder that lands inside a suppressed context sentence gets swallowed by `[CONTEXT_SUPPRESSED]` along with everything else in that sentence**, rather than surviving through to `rehydrate()`. This is a deliberate fail-safe-over-convenience trade-off consistent with "sovereignty-grade," not an oversight -- flagging here so it isn't mistaken for a rehydration bug later. Enabled by default only on `policies/finance_strict.yaml` as the reference "maximum strictness" policy; `default.yaml`/`gulf_sovereign.yaml` leave it off.

## Phase 4 — Sovereignty & agentic governance

- **"Packs" become real per-policy detector selection, not just a version-tracking label.** `DetectionPipeline.run()`'s `policy_ctx` parameter existed since Phase 1 but was dead code -- nothing ever read it. It now derives `packs` from `policy_ctx.detectors.packs` (duck-typed, no hard `Policy` import, to avoid a core/detect -> core/policy dependency) when `packs` isn't passed explicitly, and `_detect_and_repair` only runs the regex-style detector(s) named in that list. Default behavior for any existing caller that passes neither is unchanged (`("base_en",)` only) -- this was a real gap flagged by `gulf_sovereign.yaml`'s own prior "Phase 1 stub" note ("this policy currently only activates the base_en pack"), now closed.
- **gulf_ar checksum honesty tiering -- not all 7 ID types get the same confidence treatment, and none of it is faked.** Researched each format's public checksum spec before implementing (via web search + fetching reference implementations) rather than inventing one:
  - **Confirmed against a public reference implementation:** Saudi National ID / Iqama (Luhn-style even-index-doubling, `alhazmy13/Saudi-ID-Validator`) and Kuwait Civil ID (weighted mod-11, `prakhar.me`'s writeup). These get a real `checksum_valid: True/False` and a large confidence swing (+0.4 / -0.25 from a 0.55 structural baseline).
  - **Best-effort/unofficial:** UAE Emirates ID's "Luhn variant" is a community-reverse-engineered algorithm that its own author documents as rejecting some real, valid Emirates IDs -- the UAE government has never published an official spec. Still applied (nudges confidence, doesn't hard-gate), but flagged in `ids.py`'s docstring and here so nobody mistakes it for government-confirmed.
  - **No public checksum found at all -- structural pattern + Arabic cue-word boosting only:** Qatar QID, Bahrain CPR, Oman civil number. `checksum_valid` is `None` (not `False`) for these, distinguishing "didn't check" from "checked and failed." Base confidence is intentionally low (0.3-0.5) since an 8-12 digit structural pattern alone collides easily with random numbers; `morphology.has_id_cue_nearby` (an Arabic light-stemmer cue-word lookup) is the main lever available to raise confidence for these three.
- **`gulf_sovereign.yaml` gets a `GOV_ID` confidence floor (`min_confidence: 0.6`, `below_min_confidence_action: PRESERVE`) it didn't have before**, mirroring how `default.yaml` already gates PERSON/ADDRESS. Necessary precisely because of the honesty tiering above: a structural-only Oman/Qatar/Bahrain match with no cue word sits around 0.3-0.5 confidence, and without a floor, this policy's blanket `GOV_ID: BLOCK` would hard-reject content on what's really a coin-flip-odds false positive.
- **Arabic-Indic/Eastern Arabic-Indic digit folding needed no offset-remapping machinery**, unlike `core/detect/normalize.py`'s general normalization pass -- because `str.translate` with a 1:1 character mapping never changes string length or character positions, a span found against the digit-folded copy is already valid against the original text's offsets. `RawSpan.text` is still sliced from the ORIGINAL (unfolded) text so the emitted span preserves whichever digit script the source document actually used.
- **`morphology.py` is a light Arabic stemmer (Larkey Light10-style prefix/suffix stripping), not a full morphological analyzer**, and is scoped to exactly one job: normalizing a nearby word before checking it against a small ID-related cue-word set (هوية/بطاقة/اقامة/...). Not intended as a general Arabic NLP utility.

- **mcp-firewall's tool-argument rules are a distinct, separate schema from core/policy's SpanLabel-keyed `Policy`, not an extension of it.** `tools.<tool>.args.<arg>: BLOCK|REVERSIBLE|PRESERVE` (rules.py's `ToolFirewallConfig`) assigns an action per (tool, argument-name) pair directly, applied to the WHOLE argument value -- a different axis than "which action applies to this detected PII label." `core/detect`'s `DetectionPipeline` is still what actually finds the spans (scanner.py); the tool-arg rule only decides what to do with what's found. Unconfigured (tool, arg) pairs fail open to PRESERVE (still scanned for audit, never blocking/rewriting) -- an unlisted tool shouldn't silently stop working.
- **BLOCK means "this argument must be completely clean of detectable PII/secrets, or the whole call is refused" -- not "always refuse this tool."** The example `tool_rules.yaml` sets `bash.command: BLOCK`, so a clean `ls -la` still executes; only a command containing something core/detect flags gets refused. REVERSIBLE tokenizes detected spans in place using the exact same `vault.session_tokens` mechanism as `gateway/pii.py` (same deterministic per-session token IDs), so a value tokenized via a tool call and via a chat turn in the same session collide on the same token -- verified manually (identical `[PII_TOKEN_...]` for "a@b.com" via both `web_search.query` and a plain chat message in the same session).
- **mcp-firewall shares gateway/config.py's exact env var names for Valkey/vault** (`VALKEY_HOST`, `VAULT_BACKEND`, `PII_VAULT_STORAGE_PATH`, `SESSION_TOKEN_SECRET`, ...) rather than inventing its own -- deliberate, so a co-deployed gateway+mcp-firewall pair share one vault namespace per session_id out of the box, not two disconnected ones.
- **Client-facing SSE transport is spec-correct MCP (verified against the published spec + reference Python SDK shape); the upstream-facing leg is a deliberate simplification** -- a single synchronous `POST {MCP_UPSTREAM_URL}/messages` expecting the JSON-RPC reply directly in the response body, not a second bidirectional SSE session. A real MCP tool-execution node that itself only speaks SSE (not a synchronous HTTP endpoint) is NOT supported by this proxy as shipped. Verified end-to-end against a real running uvicorn instance of `proxy.py` (not just unit-level): the SSE handshake (`event: endpoint` / `data: /messages?session_id=...`), a BLOCK case (bash.command containing an email never reached the mock upstream, error relayed over SSE instead), and a REVERSIBLE case (mock upstream received the tokenized placeholder, never the raw email, and its echoed reply was correctly relayed back over the original SSE stream) -- see this session's manual verification, no automated test written per the Phase 3/4 "no tests" instruction.
- **Firewall construction (Valkey connection, vault store, pipeline) is deferred to a FastAPI `lifespan`, stored on `app.state`, not built at module-import time.** An earlier draft built it as a module-level global, which meant `import proxy` itself required a live, reachable Valkey connection -- this both mirrors `gateway/app.py`'s established lifespan pattern and makes the app constructible/testable without a running Redis (a test can override `app.state.firewall`/`app.state.settings` before the app is otherwise used).
- **Session state (`_sessions`: session_id -> asyncio.Queue) is a single-process in-memory dict on `app.state`, not Valkey-backed.** Same "real, correct, single-process; multi-worker-safe session routing is a later hardening item" tradeoff as `filescan-worker/jobs.py`'s job store and core/vault's original SQLite-before-Postgres precedent -- not pulled into scope for this phase.

- **`PostgresKeyStore` lands in its own file (`gateway/auth/postgres_key_store.py`), not as a class inside `gateway/auth/store.py` where its `NotImplementedError` stub previously lived.** Matches `core/vault/storage/`'s established sqlite/postgres-as-separate-files convention; the two `KeyStore` implementations are fully independent (each has its own `_row_to_key`, no cross-import) rather than sharing a private helper, same tradeoff `core/vault/storage/postgres_store.py` already made against `sqlite_store.py`.
- **`scripts/migrate_sqlite_to_postgres.py` copies rows as-is, never touching plaintext.** Vault rows are already encrypted (nonce/ciphertext/sealed_dek) and stay decryptable by whichever `VaultCrypto` master key was active at write time -- the migration is a pure SQL-level copy, `ON CONFLICT DO NOTHING` for idempotent re-runs. **Discovered while testing against this repo's own real `pii_vault.sqlite`: it predates the G11 `expires_at` column entirely** (the same pre-existing-dev-database scenario `SqliteVaultStore._ensure_expires_at_column()` already defends against on open) -- a hardcoded `SELECT ..., expires_at FROM vault_entries` would have crashed on this exact file. Fixed by checking `PRAGMA table_info` first and substituting a literal `NULL` when the column is absent. Verified the read-side (SQLite column detection + row iteration) against both real on-disk files in this repo (`pii_vault.sqlite`, `gateway_keys.sqlite`) with a fake `psycopg` connection recorder standing in for the write side.
- **Neither `PostgresKeyStore` nor the migration script's write side was exercised against a live Postgres server** -- no Postgres server binary or Docker access available in this environment (confirmed: no `docker` daemon permission, no local `postgres`/`initdb`/`pg_ctl`, only the `psql` client). The SQL/psycopg3 call shape is written to exactly match `core/vault/storage/postgres_store.py`'s already-established (and, per that file's own tests, working) pattern, but this is a real, flagged gap rather than a claimed-verified one -- run `docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d postgres` and re-verify before relying on this in production.

- **`bench/harness.py`'s four "vendor baseline" runners are explicitly-labeled mocks, never real API calls** -- this environment has no API keys/network access/license for Portkey, LiteLLM, Lakera, or Protecto, and even with those, treating a live third-party service's numbers as directly comparable to a synthetic local benchmark would be misleading regardless. Each mock is a small, independent stand-in reflecting only that product's PUBLIC POSITIONING (Portkey/LiteLLM: LLM gateways/routers, not PII/safety products, so their mocks do zero content scanning; Lakera: prompt-injection-focused, so its mock runs a naive independent keyword heuristic only; Protecto: PII-tokenization-focused, so its mock runs a narrow 3-pattern regex only) -- documented prominently in the module docstring, the `"caveat"` field written directly into `bench/results/latest.json` itself (not just an external README), and the rendered Markdown table. Latency/throughput numbers ARE real (every runner's own code actually executes and is timed); only each mock's detection *logic* is an illustrative simplification.
- **Fixed a real accounting bug found while building this: MonoAI's `RunnerOutcome.blocked` conflated PII-triggered blocks (e.g. a CREDIT_CARD hitting `policies/default.yaml`'s BLOCK rule) with injection-triggered blocks, corrupting the injection false-positive count** (a PII-bearing, non-injection payload that got blocked for its credit card was miscounted as an "injection false positive"). Fixed by splitting `RunnerOutcome` into `blocked` (overall accept/reject) and a separate `injection_flagged` (specifically: did the injection detector fire) -- the harness's injection TP/FP/FN/TN accounting now keys off `injection_flagged` only. Dropped MonoAI's measured injection-FP count on the synthetic set from 11 to the correct 3 once fixed.
- **Synthetic payload set reuses the existing `bench/corpora/en_pii.jsonl` + `injection.jsonl`** (already-existing, programmatically-generated starter corpora -- same size/curation caveat as the rest of `bench/`, see above) rather than generating a new one, so every runner is scored against the exact same, already-reviewed ground truth the rest of the bench suite uses.

## Step 5 — Wazuh-style manager/agent split (agent registry, /enroll, /ingest, /policy, /heartbeat + standalone `agent/`)

- **No pre-existing edge-agent / offline-buffer / auto-sync code existed to extend — this was built from scratch, and the audit said so.** The Step 1 brief assumed dormant "edge agent" and "offline buffering" code plus a runtime "SENTINEL Tier 0/1/2" tiering to generalize. A full grep (`agent`, `buffer`, `replay`, `reconnect`, `queue`, `sentinel`) found none: "SENTINEL" is only the name of the absorbed upstream submodule (now `core/detect/`), and the only "Tier N" language is the prompt-injection judge's Tier 2/2.5 (`gateway/pii.py`), a different axis from PII span detection. So the agent's buffer is a **new** implementation, not a generalization of existing simulated-local code.
- **Agent↔manager crypto uses PyNaCl authenticated `Box`, NOT the vault's `SealedBox`.** `core/vault/crypto.py`'s `SealedBox` is anonymous-sender — fine for sealing a DEK to the vault, useless for `/ingest` which must *verify* the sender is the agent it claims to be. New `core/vault/box.py` wraps `Box` (agent privkey + manager pubkey); opening the box IS the sender authentication (`/ingest` returns 401 if the envelope wasn't sealed with the registered agent's key). Reuses the existing PyNaCl/cryptography deps — no new crypto dependency.
- **The manager's agent-channel keypair is deliberately SEPARATE from the vault master key**, stored in Valkey under `monoai:agent_channel:manager_key` (distinct from `sentinel:pii_vault:master_key`), via a load-or-create helper (`gateway/agents/keys.py`) that mirrors `core/audit/signing.py`. The agent channel and the PII vault are different trust domains; sharing key material would blur the security boundary the brief made a hard constraint.
- **The agent NEVER holds any manager/vault key material — enforced structurally, not by convention.** The standalone `agent/` generates its own X25519 keypair on its own host (`agent/identity.py` → `vault.box.generate_keypair`), stores the private key `0600`, and only ever additionally holds the manager's *public* key (from the `/enroll` response). `agent/agent_config.py` has **no** Valkey/vault/manager-DB knobs — there is no config path by which manager key material could reach the agent.
- **`agent/`'s settings module is `agent_config.py`, not `config.py`.** The repo installs every workspace member's modules as FLAT top-level packages (editable `.pth` adds each member dir to `sys.path`). `agent`'s `.pth` sorts alphabetically FIRST, so an `agent/config.py` would shadow `gateway/config.py` for the whole test process and break the gateway suite (`from config import Settings`). Renaming to `agent_config` sidesteps the one real collision; the agent's other module names (`identity`, `buffer`, `client`, `enroll`, `sentinel`, `policy_cache`, `runner`) are unique across the workspace.
- **Agent events fold into the EXISTING hash-chained audit log — no parallel store.** `/ingest` decrypts a sealed batch of `AgentEventPayload`s and appends each as a normal `AuditRecord` via the same `AuditChain.append` every gateway path uses. Added ONE optional field, `agent_id: str | None = None`, to `contracts/audit.AuditRecord` for provenance; it defaults `None` so every manager-produced record and existing chain is unchanged. `AgentEventPayload` carries only labels/counts/ids/timings — invariant #1 (no raw PII in the audit surface) holds exactly as it does for `AuditRecord`.
- **`event` reused the existing `AuditRecord` Literal (`completed`/`blocked`) rather than adding agent-specific event types** — an agent detection that redacts is `completed`, one that hits a BLOCK rule is `blocked`, same semantics as the gateway's own request path. No enum change.
- **Policy `/policy` returns YAML + the manager's authoritative content-hash `version`; the agent caches both and does NOT re-hash its own serialization.** `PolicyStore` keeps only parsed `Policy` objects (version = content-hash of the *source* file text), so the endpoint re-serializes the policy body to YAML for the agent to enforce locally — but the agent stores the manager's `version` string verbatim (`agent/policy_cache.py`) and reports it on `/heartbeat`/`/ingest`. Re-hashing the agent's own YAML rendering would produce a different string and make every agent look permanently drifted. Drift = agent's held version ≠ manager's `latest_version(policy_id)`.
- **Offline buffer is a durable append-only SQLite queue (`agent/buffer.py`), modeled on `core/vault/storage/sqlite_store.py`'s shape but a different domain (ordered outbound queue, not encrypted KV) — so a new implementation, not a reuse.** Ordering is `INTEGER PRIMARY KEY AUTOINCREMENT` (monotonic, survives restart) + WAL; replay is oldest-first and rows are `ack`'d (deleted) ONLY after the manager confirms receipt, so a mid-flight failure never drops or duplicates events. Verified end-to-end in `tests/integration/test_agent_end_to_end.py` (offline → buffer 3 → reconnect → ordered replay, chain still `verify()`s).
- **Agent status (online/offline) is DERIVED on read from `last_seen` vs a freshness window (`heartbeat_interval × missed_beats`), not swept by a background job.** A crashed agent that stops beating reads as `offline` without a sweeper; `revoked` is terminal (a heartbeat can't resurrect it).
- **`agent/` is a new workspace member (`agent/pyproject.toml`), following the exact `filescan-worker/` / `mcp-firewall/` per-surface pattern — not a new project structure.** It reuses `monoai-core` as a library (same `DetectionPipeline` for Tier 0/1); Tier 2 (LLM injection judge) stays manager-side per scope.
- **Test transport: the agent's `ManagerClient` takes an injectable httpx client so tests drive a real manager app in-process via `TestClient`** (a sync `httpx.Client` that bridges to the ASGI app — `httpx.ASGITransport` is async-only and can't back a sync client). Production passes only `manager_url`.
- **Explicitly OUT of scope this pass, flagged not attempted:** Windows agent, full Wazuh parity (FIM, rootkit detection), and TEE/TPM attestation for the agent's private key (it's a `0600` file today). The `observe()` method on `AgentRunner` is the ingestion seam a real deployment wires to a traffic tap / file watcher; that host integration is also out of scope.

## Step 6 — Type-labeled PII placeholders (`<{LABEL}_PII_{id}>`) + variable-width streaming

- **Surface format only changed; `make_token_id` (the deterministic per-session HMAC) is byte-for-byte unchanged.** Placeholders went from the opaque fixed-width `[PII_TOKEN_xxxxxxxxxx]` (22 chars) to type-labeled `<{LABEL}_PII_xxxxxxxxxx>` for all 14 `SpanLabel`s (EMAIL, PHONE, CREDIT_CARD, GOV_ID, SECRET, IP_ADDRESS, DATE_TIME, USERNAME, PERSON, ADDRESS, ORG, TITLE, DEMOGRAPHIC, MISC). The 10-hex `token_id` is still embedded verbatim — rehydration reverses purely by that id (`vault.get(session_id, token_id)`), so the vault reverse-lookup path is untouched; the label is cosmetic and discarded on rehydrate (the whole matched placeholder is replaced with the stored value).
- **No value-structure preservation, by explicit instruction.** An earlier draft this session preserved email domains (`pii_<id>@gmail.com` for common webmail); that direction was superseded and reverted. The final scheme reveals only the entity TYPE, never real domains or digit groupings — uniform across every type.
- **Deliberate, flagged tradeoff: the type label is now visible to the model/provider.** `<CREDIT_CARD_PII_...>` tells the upstream provider a credit card was present — a mild info disclosure the fully-opaque token didn't make. Accepted as the point of the change (type-labeled tokens read more naturally and are less jarring to rehydrate).
- **Single source of truth for the wire format.** `core/vault/session_tokens.py` now owns `format_token(token_id, label)`, the compiled `TOKEN_RE` (`<[A-Z][A-Z_]*_PII_([0-9a-f]{10})>` — the `_PII_` sentinel + fixed 10-hex id disambiguate labels that themselves contain underscores like CREDIT_CARD/GOV_ID/IP_ADDRESS/DATE_TIME under regex backtracking), and `MAX_TOKEN_LEN` (derived from the `SpanLabel` enum so it stays correct if labels change). `gateway/pii.py` and `gateway/streaming.py` both import these instead of each maintaining their own regex/constants.
- **Streaming: kept the closed-form holdback, just sized it to the LONGEST possible token (option (b), the simpler of the two offered).** `MAX_TOKEN_LEN = 28` (`<CREDIT_CARD_PII_…>` / `<DEMOGRAPHIC_PII_…>`), so `HOLDBACK = 27`. An incomplete token of ANY type is at most `MAX_TOKEN_LEN − 1` chars, so a single holdback of that size still bounds everything in-flight — no per-type partial-match state machine needed (the module docstring notes they deliberately avoided one). The existing `_split` machinery (find last opener before the naive split; include a straddling complete token whole, else pull back before an incomplete one) is unchanged except: anchor char `[` → `<`, the constant `22 → 28`, and the shared `TOKEN_RE`. Both documented streaming bugs (off-by-one at completion, cross-`_process` fragmentation) stay fixed.
- **Verified the variable-width streaming path directly**, beyond the ported tests: a 28-char CREDIT_CARD token streamed one char at a time, a token at the very end of the stream (final flush), adjacent different-type tokens, back-to-back tokens 1-char-at-a-time, and unrelated literal `<` characters (`5 < 10`, `x<y`) — all rehydrate correctly with no fragmentation or stranding. Added two of these as permanent regression tests (`test_longest_token_type_streamed_one_char_at_a_time`, `test_unrelated_angle_bracket_does_not_break_stream`).
- **`rehydrate`'s lenient fallback extended to angle-bracket mangling.** The strict `TOKEN_RE` handles verbatim echoes; the fallback (for models that rewrite `<EMAIL_PII_id>` as `<EMAIL PII id>` / `(PERSON_PII id)`) now also matches `<…>` wrappers in addition to `[…]`/`(…)`.
- **Tests: baseline 232 passed / 4 pre-existing failures; after: 234 passed / same 4 failures.** The four format-encoding tests (`test_streaming`, `test_output_scan`, `test_chat_orchestrator`, `test_multiturn`) were updated to the new surface — semantic assertions (round-trip restores the real value, provider never sees raw PII, no leak, no fragmentation, hallucinated tokens surface as review_required) are unchanged; only the literal token strings moved to `<…_PII_…>`. `test_multiturn` needed no change (it asserts on `make_token_id` only, which is unchanged).

## Step 7 — Audit forwarding (peer gateway -> manager gateway) + canonicalization fix

### The pre-existing break this surfaced (fixed first)

- **Adding `agent_id` to `AuditRecord` (Step 5) had silently broken verification of every already-written audit record.** `_canonical_json` uses `model_dump()`, which is coupled to the pydantic schema, so once the field existed every historical record serialized with a `"agent_id": null` key that wasn't present when its hash was computed. The live 108-record `gateway_audit.jsonl` reported `TAMPERED` -- record 0 recomputed to `f5d3bf59...` vs its stored `05a5d0f8...` purely from the extra null key. This was my error in Step 5 (I reasoned "tests build fresh chains so it's safe" and dismissed the persisted log as a dev artifact; it was the real audit log). Found by running `verify_audit_chain.py` as the first step of this task.
- **Fix: `_canonical_json` omits None-valued provenance fields** (`_OMIT_WHEN_NONE = ("agent_id", "origin_gateway")`). "Key absent" (old record) and "key present but null" (new local record) now produce byte-identical canonical JSON, so adding an optional provenance field is backward-compatible by construction; a field that IS set stays fully hash-covered/tamper-evident. Pinned by `tests/unit/test_audit_chain_compat.py`, including a guard that any future None-defaulted optional field must be registered in `_OMIT_WHEN_NONE`.
- **`scripts/rehash_audit_chain.py`** re-chains a JSONL log under the current canonical form: a no-op for records that already match, recomputing only the ones written in the broken window. Refuses to run on signed logs (can't re-sign), backs up first (`.bak-<ts>`), writes atomically, and aborts unless the result verifies. Run against the live log: 106 records unchanged, 2 re-hashed, `verify_audit_chain.py` now reports `OK`.

### Forwarding design

- **`FanoutSink` + `ForwardingSink`, hooked in as an ADDITIONAL sink -- `AuditChain.append()` and every call site are untouched.** `_build_audit_sink` wraps the existing primary sink in a `FanoutSink([primary, forwarding])` only when `MONOAI_AUDIT_FORWARD_URL` is set. Local durability stays first and fail-closed; the forwarder is a best-effort secondary whose exceptions `FanoutSink` swallows.
- **`write()` does ZERO network I/O -- the request-path guarantee is structural, not best-effort.** `AuditChain.append()` is called INLINE from the async orchestrator (`orchestrator.py:227/269/288/389`), so a synchronous POST would stall every chat request for the HTTP timeout whenever the manager is down (the same latent bug `AUDIT_SINK=webhook` has today). Instead `write()` enqueues to a durable local SQLite queue and signals a background daemon thread that does delivery. "Attempt immediately" is honoured (the worker is woken on every write, not just on the interval) while manager downtime is provably isolated from the request path. A regression test asserts 20 writes against a 5s-hanging manager complete in <0.5s.
- **`SqliteForwardQueue`**: append-only, `seq INTEGER PRIMARY KEY AUTOINCREMENT` (monotonic, survives restart) + WAL, threading.Lock (request thread appends while the worker thread drains the SAME connection). Records are `ack`'d (deleted) only after a confirmed 2xx; the drain STOPS at the first failure rather than skipping ahead, so order is preserved.
- **Manager side: `POST /v1/admin/audit/ingest` in `api/admin.py`** (least-invasive: router already mounted, `_check_admin` already there). Appends the forwarded record into the manager's OWN `AuditChain` -- shows up in the existing Audit Log with no parallel store. Trust model is a shared admin key (`MONOAI_AUDIT_FORWARD_ADMIN_KEY` = the manager's `MONOAI_ADMIN_KEY`), deliberately simpler than the agent-enrollment keypair flow: this is trusted gateway-to-gateway, not the untrusted-remote-source case `api/agents.py` exists for.
- **`async def` on the ingest handler is load-bearing.** `AuditChain.append()` is a read-modify-write of the running `last_hash` and is NOT thread-safe. As an async handler it runs on the event loop with no await between read and write, so concurrent forwarders can't interleave and corrupt the chain; a plain `def` would be handed to FastAPI's threadpool and race.
- **Idempotent ingest (`gateway/audit_dedupe.py`).** Delivery is at-least-once (a lost response after the manager committed => the sender resends), so an already-seen `record_id` is answered `200 {"duplicate": true}` and not re-appended -- otherwise a compliance log would get duplicate entries, and with `AUDIT_SINK=postgres` the `record_id PRIMARY KEY` would raise -> 500 -> the sender retries the same record forever, a poison pill blocking its whole ordered queue. Persisted (survives manager restart). Verified end-to-end with a transport that drops the first response and confirms exactly-once append on retry.
- **`origin_gateway` distinguishes forwarded from local records** in the Audit Log: `None` for locally-generated, the sender's `MONOAI_GATEWAY_ID` for forwarded. The sender stamps it on the FORWARDED copy only -- its own local copy stays `origin_gateway=None` (locally it *is* local). The ingest endpoint refuses a record without it rather than silently filing a peer's record as its own.
- **Restart-safety bug caught + fixed:** the `initial_last_hash` bootstrap uses `isinstance(sink, JsonlSink/PostgresSink)`. Wrapping the sink in `FanoutSink` would make neither branch match, silently resuming with `prev_hash=None` and breaking the chain at every restart. `app.py` now builds the primary sink, bootstraps from IT, then wraps -- and closes the audit sink on shutdown (sinks were never closed before; that also leaked `WebhookSink`'s httpx client and would now leave the forwarder thread running past shutdown).
- **Scope respected:** no change to `chat.py`, `pii.py`, or `orchestrator.py` -- purely additive to the audit/sink layer. Off unless `MONOAI_AUDIT_FORWARD_URL` is configured.
- **Tests: 234 -> 261 passing (+27; canonicalization compat, forward queue, sink non-blocking/retry/order, end-to-end forward->manager chain + dedupe). Same 4 pre-existing failures, no regression.**

## Step 8 — Virtual-key federation (forward keys to manager + bidirectional revoke)

Applies the audit-forwarding pattern to virtual-key create/revoke so a key created on a forwarding gateway appears in the manager's Users tab. Additive, parallel to audit forwarding -- the audit path (ForwardingSink/forward_queue/audit ingest) is NOT touched.

- **Reused `SqliteForwardQueue` and `SqliteIngestDedupe` unchanged.** Both are already payload-agnostic (an id string + opaque JSON), so the key stream just instantiates them with separate DB paths -- no `event_type` flag added to the queue, no generalization of audit code. This was the least-invasive option (vs. adding a payload-type column to a store the audit path depends on).
- **New `KeyEventForwarder` (gateway/key_forwarder.py) is a self-contained sibling of `ForwardingSink`, not a shared-base refactor** -- deliberately, to honour "don't touch the audit path." It carries the same guarantees (enqueue does zero network I/O and never raises; background worker drains oldest-first, stops at first failure to preserve order, acks only on 2xx; at-least-once + dedupe). It adds one thing ForwardingSink lacks: **per-event target URL** (via a `_target_url` in the payload), because the reverse direction sends each revoke to a different origin gateway.
- **Bidirectional, per the user's decision A.** Forward (origin→manager): create/revoke hooks in `api/admin.py` enqueue a `KeyForwardEvent` → manager's `POST /v1/admin/keys/ingest` mirrors it into its own KeyStore via a new `add_forwarded_key()` (create) or `revoke()` (revoke), tagged `origin_gateway`. Reverse (manager→origin): revoking a *forwarded* key on the manager optimistically revokes the local copy AND enqueues a `KeyRevokeEvent` to the origin's `POST /v1/admin/keys/revoke-ingest`, which applies `revoke()` to the origin's own store. **The reverse-ingest deliberately does NOT re-forward** -- that would loop back to the manager, which already marked its copy. Loop broken by construction.
- **`create_key()` can't be reused for ingest** -- it mints a fresh key_id/hash. A forwarded key must preserve the origin's identity (so dedupe and the Users-tab key_id line up), so a new `add_forwarded_key(VirtualKey)` (INSERT-OR-IGNORE on key_id) was added to the KeyStore protocol + both SQLite and Postgres impls, plus `get_by_id()`.
- **Decision B: forwarded keys are visibility-only.** `authenticate()` (auth/middleware.py) now rejects any key with `origin_gateway` set -- a one-line guard. Storing Rahul's key_hash for display would otherwise silently make his key valid against the manager's gateway; this keeps access scoped to the owning gateway. Flagged and confirmed with the user because it's an access-control implication, not just logging.
- **Schema migration, in place.** `virtual_keys` gains `origin_gateway` + `origin_callback_url`. `SqliteKeyStore._ensure_federation_columns()` (mirrors vault's `_ensure_expires_at_column`) and Postgres `ADD COLUMN IF NOT EXISTS` upgrade an existing `gateway_keys.sqlite`/table in place -- verified against a hand-built pre-migration DB.
- **Reverse propagation needs the origin's address + auth (genuinely new).** New optional `MONOAI_GATEWAY_CALLBACK_URL`: a gateway's own reachable base URL, advertised in its forward events; the manager stores it (`origin_callback_url`) and pushes reverse revokes there. Unset ⇒ the key is visible on the manager but not remotely revocable, and the revoke response says so honestly (`propagated: false`) rather than pretending. Reverse auth uses the shared federation admin key (`MONOAI_AUDIT_FORWARD_ADMIN_KEY`), the same symmetric-trust assumption audit forwarding already makes.
- **The keys-ingest URL is DERIVED from `MONOAI_AUDIT_FORWARD_URL`** (`Settings.key_forward_url` swaps the path to `/v1/admin/keys/ingest`) -- no new user-facing URL var. Key federation is gated on `MONOAI_AUDIT_FORWARD_ADMIN_KEY` being set (federation participant): a non-participant gets no threads or state files.
- **Postgres path mirrored but NOT tested live** (no Postgres server available here, same gap flagged in earlier steps). The psycopg call shapes match the existing working patterns; run against a real Postgres before relying on the `postgres` key-store backend with federation.
- **Tests: 261 → 273 passing (+12: 8 KeyEventForwarder unit tests, 4 bidirectional end-to-end across two live gateway apps -- create→forward→Users-tab, forwarded key rejected by auth on manager but valid on origin, manager-side revoke propagating back to deactivate on the origin, idempotent ingest, no-callback-url honesty). Same 4 pre-existing failures, no regression.** New `.sqlite` state files are already covered by the `*.sqlite` gitignore rule.

## Step 10 — Provider/model config sync (manager -> instance, DOWNWARD)

Providers/models flow the OPPOSITE direction from audit/key forwarding: the manager is the single source of truth and forwarding instances pull its registry. Additive; does not touch the audit/key-forwarding code paths.

- **PULL, not PUSH (recommended + chosen).** Instances already never need to be reachable BY the manager (audit/keys go instance->manager; the only manager->instance path, reverse key-revoke, is best-effort). PUSH would invert that and require a manager-side registry of every instance's address + Box pubkey. PULL is self-healing (a new/rebooted instance catches up on its next poll) and makes the sealed-key handshake a clean request/response (the instance presents its pubkey in the poll). No new instance registry, no `callback_url` dependency.
- **Manager-exclusive (user's choice).** A syncing instance's registry mirrors ONLY the manager's set; `replace_registry` deletes the whole local providers/models tables and re-inserts the manager's, stamped `origin_gateway=<manager_id>`. Local provider/model add+delete routes return 409 on a syncing instance (`_reject_if_provider_synced`, keyed on `app.state.provider_sync` being present) -- otherwise a local edit would be silently wiped on the next poll.
- **API keys never traverse the wire in plaintext (requirement 1).** Provider keys are encrypted at rest under each gateway's OWN VaultCrypto (its own Valkey master key), so an instance cannot decrypt the manager's ciphertext regardless of transport -- confirmed. The manager therefore DECRYPTS each key and RE-SEALS it to the requesting instance's X25519 pubkey using the existing authenticated Box (core/vault/box.py, the Wazuh-agent primitive -- no new crypto). The instance opens it with its own privkey and re-encrypts under its own VaultCrypto on insert. Only `api_key` values are sealed; non-secret config (names, base_urls, model ids, key_last4) is plain JSON under the shared-admin-key auth. A test asserts the raw sync response never contains the plaintext key.
- **Live apply, no restart (requirement 3).** `DynamicProviderRouter.resolve()` already reads the store per request, so writing reconciled rows takes effect immediately; the sync calls the new `DynamicProviderRouter.invalidate()` after a change to drop cached provider adapters (a changed base_url/key must not keep serving from a stale cached adapter). Change is detected by comparing a provider/model signature (key_last4 catches key rotation), so adapters aren't dropped every poll when nothing changed.
- **Fail-open / resilience.** A failed poll (manager down, malformed response, a sealed key that won't open) returns without touching the local registry -- the instance keeps serving last-known-good config. The poller is a background daemon thread that never blocks or crashes the gateway; closed on shutdown.
- **Config reuses audit-forwarding env (requirement: opt-in, no new required vars).** Enabled when `MONOAI_AUDIT_FORWARD_URL` + `MONOAI_AUDIT_FORWARD_ADMIN_KEY` are set; the sync URL is DERIVED (`provider_sync_url` -> `/v1/admin/providers/sync`). Only internal knobs added: `MONOAI_PROVIDER_SYNC_INTERVAL_S` (default 60) and the instance Box key name. The instance Box keypair reuses `agents.keys.load_or_create_manager_keypair` under a distinct Valkey name.
- **Thread-safety.** `SqliteProviderStore` gained an `RLock` (re-entrant: `add_model`->`get_provider`; `resolve`->`get_decrypted_key`) guarding the shared connection, since the sync thread writes while request threads read via `resolve()` -- the same request-thread-vs-background-thread concern `forward_queue.py` solved with a lock. `origin_gateway` column added to both tables with an in-place ALTER migration (verified against a pre-migration DB).
- **Tests: 276 -> 282 (+6): pull+resolve of a synced model with the correct decrypted key; plaintext-key-never-on-wire; live apply of a manager-side change; fail-open when the manager is down; endpoint auth/pubkey validation; manager-exclusive local-add rejection. Same 4 pre-existing failures, no regression.** New `.sqlite` files are covered by the existing `*.sqlite` gitignore.
- **Postgres note:** provider registry is SQLite-only in this codebase (unlike keys/vault which have Postgres backends), so no Postgres mirror was needed.

## Step 11 — Session federation: forward per-user chat sessions instance -> manager

Extends federation so a user created on a forwarding instance shows their FULL session drill-down (Original -> Redacted -> Reply -> Rehydrated) in the manager's Users tab, not just their key. Answers the observed gap: a forwarded key appeared in the Users tab but its sessions were empty, because only audit records + key events federated -- transactions did not.

- **This carries the most sensitive data in the system (raw prompt/reply), so it's encrypted at EVERY hop:**
  * origin at rest -- `TransactionForwarder.enqueue()` encrypts the 4 text fields with THIS gateway's own VaultCrypto BEFORE they enter the durable forward queue (the queue never holds plaintext), exactly as `SqliteTransactionStore` already does on disk.
  * in transit -- at delivery the worker decrypts in memory only, then RE-SEALS the blob to the manager with the authenticated PyNaCl Box (`core/vault/box.py`, the Wazuh-agent primitive -- no new crypto). Only the manager's private key can open it. A test asserts the raw SSN/card never appear in the request body, only the sealed ciphertext.
  * manager at rest -- the manager opens the Box and re-encrypts under its OWN VaultCrypto via `transaction_store.record()`. A forwarded session is stored identically to a local one; a test greps the manager's sqlite file to confirm the plaintext isn't present.
- **PULL wasn't an option here (unlike provider sync): sessions are produced on the instance, so this is instance -> manager (UPWARD), mirroring audit/key forwarding.** Reuses `SqliteForwardQueue` + `SqliteIngestDedupe` unchanged; new `TransactionForwarder` is a self-contained sibling of the other forwarders (durable queue, background worker, oldest-first, ack-on-2xx, at-least-once + dedupe on `request_id`).
